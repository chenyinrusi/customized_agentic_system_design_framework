# 设计理念与架构框架

## 1. 设计理念

### 1.1 明确目标："让 AI 成为可信赖的系统"
核心思想不是打造一个更聪明的 AI，而是构建一个多 Agent 协作的系统，让它安全、透明、可靠地完成工作。

- **信任优先**：产品必须可预测、可审计，而不是只追求炫酷。
- **人为主控**：每一个危险操作都应经过治理、审批和人工确认。
- **透明可见**：每一个决策、工具调用和成本事件都应可观察。
- **容错性强**：当某个 LLM 服务不可用时，系统必须继续运行。
- **记忆连续**：系统应记住项目决策，而不是仅保留一次性对话。

### 1.2 这个设计的不同之处

- 许多 AI 系统都把重点放在一个单一的助手上。
- 本项目有意做成模块化：每个 Agent 有专门职责，系统负责协调它们。
- 目标不是“AI 助手”，而是“AI 驱动的工程平台”。
- 价值在于管理风险、成本和持续工程工作，而不是只是生成结果。
- 系统明确把 Agent 的“大脑”与编排、执行和工具调用分离：我们把 LLM Mode 看作大脑，而 Agent 负责任务流程和外部工具。

这种分离意味着你可以在项目的任何部分、Agent 执行的任何步骤中更换 LLM Mode，从而在保密性、成本和质量之间做更精细的权衡。

## 2. 架构框架

### 2.1 核心层次结构

架构分为几个明确的层次：

1. **编排层**
   - 控制任务生命周期。
   - 管理多个 Agent 及其依赖关系。
   - 将任务分配给合适角色。

2. **推理层**
   - 负责生成计划和任务分解。
   - 使用更强的模型处理策略和审查任务。
   - 支持基于反馈的计划修订。

3. **执行层**
   - 负责代码生成、测试、审查和知识记录。
   - 执行工具调用，例如 shell 命令、文件写入、Git 操作和浏览器操作。

4. **治理层**
   - 强制执行 RBAC 和审批策略。
   - 记录审计事件和风险元数据。
   - 控制高风险操作和预算敏感决策。

5. **可观测层**
   - 记录所有工具调用和 Agent 状态。
   - 跟踪成本、Token 使用和执行链。
   - 让系统对用户可见。

6. **记忆层**
   - 持久化项目上下文和决策。
   - 提供结构化的跨会话记忆。
   - 通过压缩和清理保持历史记录可用。

### 2.2 Agent 角色划分

系统围绕专门化角色设计，而不是一个通用助手：

- **Planning Agent**：分解目标、选择模型、编排任务。
- **Coding Agent**：编写代码、生成实现。
- **Research Agent**：搜索资料、分析架构、推荐方案。
- **Testing Agent**：编写并执行测试。
- **Review Agent**：审查代码质量、安全性和最佳实践。
- **Memory Agent**：记录项目决策、约束和历史上下文。

每个角色承担独立责任，并可并行运行。

一个关键原则是：LLM Mode 的粒度要比 Agent 更小。这意味着可以把复杂步骤交给更贵、更强的 Mode，把简单步骤交给更便宜的 Mode；也可以把数据和任务拆分给不同的 Mode，让单一 AI 公司不会看到整个项目、数据或想法。

### 2.3 多 Agent 编排模型

- 系统使用任务队列和调度器来并行执行多个 Agent。
- 依赖关系被解析，使得安全性允许时可以并行推进。
- Agent 之间通过轻量消息总线和共享记忆交换结论。
- 最终由汇总步骤聚合结果，保证一致性。

### 2.4 治理与安全原则

- **先阻断，再许可**：危险操作默认阻止，须明确审批。
- **角色感知权限**：不同 Agent 和用户角色具有不同能力边界。
- **预算感知**：模型选择受成本预算和使用阈值影响。
- **审计优先**：每次操作都记录上下文、结果和审批状态。

### 2.5 可观测与诊断

- 系统设计为支持运行时监控 Agent 状态。
- 仪表盘展示进度、Token 消耗、成本和待审批项。
- 执行链将用户请求与工具调用和 Agent 决策关联。
- 这种透明度是产品功能，而不是事后补充。

### 2.6 记忆与持续性

- 项目记忆保存决策、约束、架构笔记和风险。
- 记忆是结构化的，而不是原始聊天记录。
- 系统自动将相关记忆注入未来 Agent 提示。
- 通过老化、去重和压缩保持记忆有用且高效。

## 3. 高层产品叙事

### 3.1 解决的问题

目前多数 AI 开发工具要么：
- 是单次互动的黑箱助手；
- 要么是松散的工具集，缺乏长期项目连续性。

本产品填补了这之间的空白，提供：

- 多个专用 Agent 的并行工作能力
- 严格的操作安全和审批控制
- 透明的成本与行为跟踪
- 跨会话的持久项目记忆

### 3.2 价值主张

- 通过并行 Agent 提高交付速度。
- 通过治理控制提升自动化安全性。
- 通过可观测降低使用风险。
- 通过记忆增强项目连续性。
- 通过模型路由和回退实现实际成本管理。

### 3.3 为什么这很重要

工程工作本质上是协作、迭代和有状态的。
一个丧失上下文或失控运行的系统，并不适合生产。
这个设计旨在把 AI 变成一个受控、可测量的合作伙伴，而不是一个实验性玩具。

## 4. 差异化设计选择

### 4.1 默认多 Agent

大多数系统把多 Agent 视为可选或实验性功能。
本设计把它作为解决非平凡工程问题的默认方式。

### 4.2 治理是第一级功能

不是后续才补上治理，而是假设每个操作都有可能需要审批。
这让产品更适合团队和企业使用。

### 4.3 记忆作为产品特性

系统不只是保留对话历史。
它保存的是跨天、跨任务的关键决策和约束。

### 4.4 失败即回退

当一个 API 挂掉时，系统不是停止，而是顺着备用模型继续。
这带来更稳定的用户体验。

### 4.5 可观测性建立信任

系统假设用户应该始终知道发生了什么以及为什么发生。
这正是从实验性工具迈向可信赖工程平台的核心区别。

## 5. 如何在不公开源码的情况下表达

如果你想传达想法而不公开实现细节，可以这样描述：

- 说它是“一个多 Agent 工程平台”，而不是“一个 Python 项目”。
- 说它是“治理优先的 AI 系统”，而不是“一个库集合”。
- 说它是“项目记忆引擎”，而不是“一个数据库实现”。
- 说它是“成本感知模型路由器”，而不是“提供商回退表”。

价值在于设计模式，本质不是代码本身。

## 6. 关键设计陈述

- “本产品把 AI 视为一组协作的专家，而不是一个单一的预言机。”
- “它从第一天起就为可观测、可审计、成本感知而设计。”
- “它记住项目决策，而不是让用户反复解释上下文。”
- “它通过备用模型和提供商回退来优雅地处理失败。”
- “它默认阻止危险操作，只有在用户批准后才执行。”
